2. Revision: Why Do We Encode Categorical Data?

Real-world datasets frequently contain categorical values in the form of strings.

Example:

NameGenderCity
RahulMaleDelhi
PriyaFemaleMumbai
AmanMaleKolkata

The values: Male, Female, Delhi, Mumbai, Kolkata are strings.

Most machine-learning algorithms cannot directly perform mathematical calculations on strings. They expect numerical input.

Therefore, it is the responsibility of a data scientist to convert categorical values into a numerical representation.

The process of converting categorical data into numbers is called:

Categorical Encoding

However, we should not assign numbers randomly. The encoding method must preserve the actual meaning of the data.

3. Types of Categorical Data

3.1 Ordinal Categorical Data

Ordinal data contains categories with a meaningful order or ranking.

Example:

Poor < Average < Good < Excellent

We can assign:

Poor → 0 Average → 1 Good → 2 Excellent → 3

This numerical order correctly represents the actual relationship between the categories.

Therefore, we generally use:

OrdinalEncoder

for ordinal input features.

3.2 Nominal Categorical Data

Nominal data contains categories with no natural order.

Examples:

Male Female Red Green Blue Delhi Mumbai Guwahati Petrol Diesel CNG LPG

For example: Red = Blue = Green. No colour is naturally greater or smaller than another colour.

Similarly: Male = Female. Neither category has a higher rank.

Therefore, we should generally use:

One-Hot Encoding

4. Why Should We Not Use Ordinal Encoding for Nominal Data?

Suppose we have three colours:

Yellow Blue Red

If we use Ordinal Encoding:

Yellow → 0 Blue → 1 Red → 2

The machine-learning model may interpret these values mathematically.

It may assume:

Red > Blue > Yellow

It may also assume:

Distance between Red and Blue = 1 Distance between Blue and Yellow = 1

Some models may even interpret:

Red = 2 × Blue

However, these relationships do not exist. Colours are simply different categories. They do not have rankings or meaningful numerical distances.

Conclusion: Assigning ordinary integer values to nominal categories may create a false order. One-Hot Encoding solves this problem.

5. What Is One-Hot Encoding?

One-Hot Encoding is a technique used to convert nominal categorical values into numerical vectors.

It creates a separate binary column for every category.

Binary means that the columns contain:

0 or 1

Suppose the original data is:

Colour
Yellow
Blue
Red
Blue

There are three unique categories: Yellow, Blue, Red.

Therefore, One-Hot Encoding creates three new columns: Colour_Yellow, Colour_Blue, Colour_Red.

The transformed dataset becomes:

Original ColourColour_YellowColour_BlueColour_Red
Yellow100
Blue010
Red001
Blue010

Each row contains 1 in the column representing the category and 0 in all other category columns.

6. Vector Representation

One-Hot Encoding converts every category into a binary vector.

For example:

Yellow → [1, 0, 0] Blue → [0, 1, 0] Red → [0, 0, 1]

Each vector uniquely identifies a category. No artificial ranking is introduced.

The model does not assume Red > Blue or Blue > Yellow. All categories are treated independently.

7. Why Is It Called “One-Hot”?

Consider: Yellow → [1, 0, 0]

Only one position is active or hot. The active position contains 1. All other positions contain 0.

Therefore, it is called: One-Hot Encoding

8. Number of Columns Created

Suppose a categorical feature contains n categories. Normal One-Hot Encoding creates n new columns.

Number of categoriesNumber of OHE columns
22
33
1010
5050
500500

Therefore: Number of OHE columns = Number of unique categories

If a feature contains 500 unique categories, normal One-Hot Encoding may create 500 columns. This can significantly increase the dimensionality of the dataset.

9. Dummy Variables

The new binary columns created by One-Hot Encoding are called:

Dummy Variables

For example: Colour_Yellow, Colour_Blue, Colour_Red are dummy variables. They contain binary values: 0 and 1.

10. Dummy Variable Trap

Suppose One-Hot Encoding creates:

Colour_YellowColour_BlueColour_Red
100
010
001

Notice: Colour_Yellow + Colour_Blue + Colour_Red = 1

This relationship is true for every row. Therefore, one column can be perfectly calculated using the other columns.

For example: Colour_Yellow = 1 − Colour_Blue − Colour_Red

Therefore, the columns are not completely independent. This perfect dependency may cause a problem called:

Perfect Multicollinearity

The problem caused by keeping all dummy columns is commonly called:

Dummy Variable Trap

11. What Is Multicollinearity?

Multicollinearity occurs when two or more input features have a strong mathematical relationship.

For example: X₃ = X₁ + X₂

In this situation, one input feature can be predicted using other input features.

Ideally, input features should not be perfectly dependent on each other.

Suppose: Yellow + Blue + Red = 1. Then: Yellow = 1 − Blue − Red

Therefore, the Yellow column does not provide completely independent information.

12. Solution to the Dummy Variable Trap

If there are n categories, we can keep n − 1 columns.

Suppose there are three categories: Yellow, Blue, Red. Instead of keeping all three columns, we can remove one column (e.g., Yellow).

The remaining data becomes:

Colour_BlueColour_RedMeaning
00Yellow
10Blue
01Red

We can still identify all three categories. How?

Blue = [1, 0] Red = [0, 1] Yellow = [0, 0]
n − 1 dummy columns are sufficient to represent n categories.

13. Important Modern Clarification ⭐

Dropping one dummy column is particularly important for models such as:

  • Linear Regression
  • Logistic Regression
  • Generalized Linear Models
  • Statistical regression models

These models may be affected by perfect multicollinearity.

However, dropping a category is often unnecessary for many tree-based algorithms, such as:

  • Decision Tree, Random Forest, Extra Trees
  • Gradient Boosting, XGBoost, LightGBM

Tree-based models generally do not suffer from the dummy variable trap in the same way as linear models.

Therefore:

Linear models → usually consider drop='first' Tree models → keeping all categories is often acceptable

Do not automatically use drop='first' in every project. The choice depends on the model.

pd.get_dummies(

15. Removing the First Dummy Column Using Pandas

Use:

pd.get_dummies(
    df,
    columns=['Fuel', 'Owner'],
    drop_first=True
)

drop_first=True removes the first category from every encoded feature.

If Fuel categories are CNG, Diesel, Petrol, then Pandas may remove Fuel_CNG.

If both remaining values are zero (Fuel_Diesel = 0, Fuel_Petrol = 0), the category is CNG.

16. Pandas get_dummies() Syntax

pd.get_dummies(
    data,
    columns=None,
    drop_first=False,
    dtype=None
)

Important parameters:

ParameterMeaning
dataInput DataFrame
columnsColumns to encode
drop_firstRemoves the first category
dtypeControls the data type of encoded values

Example:

encoded_df = pd.get_dummies(
    df,
    columns=['Fuel', 'Owner'],
    drop_first=True,
    dtype=int
)

Using dtype=int creates 0 and 1 instead of False and True.

17. Why Is pd.get_dummies() Usually Not Preferred in Production Pipelines?

pd.get_dummies() is useful for: Exploratory Data Analysis, Quick experiments, Learning, Small projects.

However, in machine-learning projects, we usually need the encoder to:

  1. Learn categories only from training data
  2. Remember the learned categories
  3. Apply exactly the same transformation to test data
  4. Maintain the same column order
  5. Handle unseen categories
  6. Work inside a pipeline

Scikit-learn's OneHotEncoder is designed for these tasks.

Quick analysis → pd.get_dummies() ML pipeline → sklearn OneHotEncoder

18. One-Hot Encoding Using Scikit-Learn

from sklearn.preprocessing import OneHotEncoder

ohe = OneHotEncoder()

Suppose X_train contains: Brand, Kilometers_Driven, Fuel, Owner. We only want to encode Fuel and Owner.

# Fit and transform training data
X_train_encoded = ohe.fit_transform(
    X_train[['Fuel', 'Owner']]
)

# Transform test data
X_test_encoded = ohe.transform(
    X_test[['Fuel', 'Owner']]
)

Notice: fit_transform() is used on training data. Only transform() is used on test data.

19. Why Do We Use fit_transform() on Training Data?

During fit(), the encoder learns: Unique categories, Category order, Number of output columns.

During transform(), it converts the values into numerical vectors using the learned information.

Therefore, fit_transform() performs: Learn + Transform

20. Why Do We Use Only transform() on Test Data?

The test dataset should not teach the preprocessing model anything.

Incorrect: ohe.fit_transform(X_test) because the encoder would learn information from the test data. This can create Data Leakage.
Correct pattern:
ohe.fit_transform(X_train)
ohe.transform(X_test)

from sklearn.model_selection import train_test_split

22. Creating OneHotEncoder

Basic:

from sklearn.preprocessing import OneHotEncoder
ohe = OneHotEncoder()

Remove the first category:

ohe = OneHotEncoder(drop='first')

Return a normal NumPy array:

ohe = OneHotEncoder(drop='first', sparse_output=False)

Return integer values:

import numpy as np
ohe = OneHotEncoder(drop='first', sparse_output=False, dtype=np.int32)

23. Important Parameters of OneHotEncoder

23.1 drop

OneHotEncoder(drop='first')

It removes the first category from every encoded feature.

Example: If Fuel has CNG, Diesel, Petrol, LPG, after drop='first' one category is removed. Possible output: Fuel_Diesel, Fuel_LPG, Fuel_Petrol. Fuel_CNG becomes the reference category.

23.2 sparse_output

By default, OneHotEncoder may return a sparse matrix. A sparse matrix stores mostly non-zero values efficiently. One-Hot Encoded datasets contain many zeros.

To get a normal NumPy array:

ohe = OneHotEncoder(sparse_output=False)

Older Scikit-learn versions used sparse=False. Modern versions use sparse_output=False.

23.3 dtype

By default, encoded values may be floating-point numbers (0.0, 1.0). To get integers:

import numpy as np
ohe = OneHotEncoder(sparse_output=False, dtype=np.int32)

23.4 handle_unknown

Suppose training data contains Petrol, Diesel, CNG, but test data contains Electric. The encoder has never seen Electric. By default, this may produce an error.

Use:

ohe = OneHotEncoder(handle_unknown='ignore')

Now unseen categories will not cause an error (represented using zeros). This parameter is highly useful in real-world applications.

Recommended:
ohe = OneHotEncoder(handle_unknown='ignore', sparse_output=False)

24. Complete Scikit-Learn Example

import numpy as np
from sklearn.preprocessing import OneHotEncoder

ohe = OneHotEncoder(
    drop='first',
    handle_unknown='ignore',
    sparse_output=False,
    dtype=np.int32
)

X_train_encoded = ohe.fit_transform(X_train[['Fuel', 'Owner']])
X_test_encoded = ohe.transform(X_test[['Fuel', 'Owner']])

29. High Cardinality

Suppose a car dataset contains a Brand column with 32 unique brands (Maruti, Hyundai, Mahindra, Tata, etc.). Normal One-Hot Encoding creates approximately 32 new columns.

This is called:

High Cardinality

Cardinality means the number of unique categories in a categorical feature.

FeatureUnique valuesCardinality
Gender2Low
Fuel4Low
State28Medium
Brand100High
Customer ID1,000,000Extremely high

30. Problems Caused by High Cardinality

High-cardinality One-Hot Encoding may cause:

  1. Too many columns
  2. Increased memory usage
  3. Increased training time
  4. Slower predictions
  5. Sparse datasets
  6. Increased model complexity
  7. Possible overfitting
  8. Poor generalization for very rare categories

This increase in the number of dimensions is sometimes called: Dimensionality Explosion

31. Handling Rare Categories

Suppose: Maruti (2500 cars), Hyundai (1800), Mahindra (900), Jaguar (8), Volvo (5), Datsun (3).

Instead of creating separate columns for every rare brand, combine rare categories into:

Other

The transformed values become: Maruti, Hyundai, Mahindra, Other. This reduces dimensionality.

32. Most-Frequent Category Strategy

One common approach:

  1. Calculate category frequencies
  2. Keep the most frequent categories
  3. Replace rare categories with Other
  4. Apply One-Hot Encoding
brand_counts = df['Brand'].value_counts()

# Select categories containing at least 100 observations
frequent_brands = brand_counts[brand_counts >= 100].index

# Replace rare categories
df['Brand'] = df['Brand'].where(
    df['Brand'].isin(frequent_brands),
    'Other'
)

# Apply One-Hot Encoding
df = pd.get_dummies(df, columns=['Brand'], drop_first=True, dtype=int)

33. Complete Rare-Category Example

import pandas as pd

brand_counts = df['Brand'].value_counts()
rare_brands = brand_counts[brand_counts < 100].index

df['Brand'] = df['Brand'].replace(rare_brands, 'Other')

encoded_df = pd.get_dummies(df, columns=['Brand'], dtype=int)

Now all brands containing fewer than 100 observations are represented by Brand_Other.

34. Modern Scikit-Learn Method for Rare Categories ⭐

Modern versions of Scikit-learn can automatically group infrequent categories.

ohe = OneHotEncoder(
    min_frequency=100,
    handle_unknown='infrequent_if_exist',
    sparse_output=False
)

min_frequency=100 means categories appearing fewer than 100 times may be grouped as infrequent.

You can also use:

ohe = OneHotEncoder(
    max_categories=10,
    handle_unknown='infrequent_if_exist',
    sparse_output=False
)

This limits the number of output categories. This is cleaner than manually replacing rare categories.

35. Important Warning About Rare-Category Grouping

The frequency threshold should be learned only from training data.

Do not calculate rare categories using the complete dataset before train-test splitting.

Incorrect:

# Complete dataset used before splitting
counts = df['Brand'].value_counts()

This may allow information from the test dataset to influence preprocessing.

Better:

  1. Split the data
  2. Learn category frequencies from training data
  3. Apply the same rule to test data

Using a pipeline is the safest approach.

36. Advantages of One-Hot Encoding

  • ✅ No false ranking is introduced
  • ✅ Easy to understand
  • ✅ Easy to implement
  • ✅ Works well with nominal data
  • ✅ Compatible with many machine-learning algorithms
  • ✅ Every category receives an independent representation
  • ✅ Useful for linear models

37. Disadvantages of One-Hot Encoding

  • ❌ Creates many new columns
  • ❌ Can increase memory consumption
  • ❌ Can slow model training
  • ❌ Can cause high dimensionality
  • ❌ May cause the Dummy Variable Trap in some models
  • ❌ Not ideal for extremely high-cardinality features
  • ❌ Unknown categories require proper handling

38. When Should We Use One-Hot Encoding?

Use One-Hot Encoding when:

  • Data is nominal
  • Categories have no natural order
  • The number of categories is low or moderate
  • You do not want to introduce an artificial ranking

Examples: Gender, Colour, Fuel Type, City, Country, Department, Payment Method, Product Category.

39. When Should We Avoid One-Hot Encoding?

Avoid or reconsider normal OHE when:

  • A feature contains thousands of categories
  • The feature is almost unique for every row
  • The feature is an ID

Examples: Customer ID, Transaction ID, Email Address, Phone Number, Product ID. One-Hot Encoding IDs usually creates a huge number of meaningless columns.

40. Alternatives for High-Cardinality Features

Possible alternatives include:

  1. Rare-category grouping
  2. Frequency Encoding
  3. Count Encoding
  4. Target Encoding
  5. Binary Encoding
  6. Hash Encoding
  7. Feature Hashing
  8. Learned embeddings

These methods will be useful in advanced machine-learning projects.

43. Common Mistakes ⚠️

Mistake 1: Using Ordinal Encoding for nominal data

Wrong: Red → 0, Blue → 1, Green → 2. This creates a false ranking.

Mistake 2: Fitting the encoder on test data

Wrong: ohe.fit_transform(X_test)
Correct:
ohe.fit_transform(X_train)
ohe.transform(X_test)

Mistake 3: Fitting before train-test splitting

Wrong:
ohe.fit_transform(X)
train_test_split(...)
Correct:
train_test_split(...)
ohe.fit_transform(X_train)
ohe.transform(X_test)

Mistake 4: Not handling unseen categories

Use: handle_unknown='ignore'

Mistake 5: One-Hot Encoding unique IDs

Avoid encoding: Customer_ID, Transaction_ID, Phone_Number

Mistake 6: Using drop='first' without considering the model

It is useful for linear models but is not compulsory for every algorithm.

45. Interview Questions 🎯

Q1. What is One-Hot Encoding?

One-Hot Encoding converts nominal categorical values into binary vectors by creating one binary column for every category.

Q2. Why is One-Hot Encoding used?

It converts nominal categorical data into numerical data without introducing an artificial order.

Q3. What is a dummy variable?

A dummy variable is a binary column containing 0 or 1 created during One-Hot Encoding.

Q4. What is the Dummy Variable Trap?

The Dummy Variable Trap occurs when all dummy columns are retained and one dummy column can be perfectly predicted from the others, creating perfect multicollinearity.

Q5. How can we avoid the Dummy Variable Trap?

If there are n categories, keep only n − 1 dummy columns. In Scikit-learn: OneHotEncoder(drop='first')

Q6. What is high cardinality?

High cardinality means a categorical feature contains many unique categories.

Q7. How can we handle high-cardinality data?

Possible methods: Group rare categories into Other, Keep only frequent categories, Frequency Encoding, Target Encoding, Hash Encoding, Embeddings.

Q8. What does handle_unknown='ignore' do?

It prevents errors when new categories appear during transformation.

Q9. Why is OneHotEncoder preferred over pd.get_dummies() in ML pipelines?

OneHotEncoder learns and remembers categories, applies consistent transformations, handles unknown categories and works with Scikit-learn pipelines.

Q10. What is the difference between fit_transform() and transform()?

fit_transform(): Learns categories + transforms data.
transform(): Uses already learned categories to transform data.
